# KTMB Pulse — Agent Guidelines

## Project Identity
This is **KTMB Pulse** (Crowd & Trend), a crowd-forecast and live-tracking app for Malaysia's KTM railway network. Two platforms: Android (Kotlin/Compose, primary) and HTML prototype (docs/, secondary).

## Repository Overview
- **Language**: Kotlin (Android) + Python (data pipeline) + JavaScript (Cloudflare Worker)
- **Build system**: Gradle (Kotlin DSL) for Android; no build system for Python/JS
- **Package**: `com.ktmb.crowdtrend`
- **CI/CD**: GitHub Actions for data pipeline only; no Android CI
- **Deployment**: APK via Gradle, Cloudflare Worker via Wrangler
- **GitHub**: `abinmo3/ktmb-dashboards` (public)

## Architecture Summary

```

├── android/          # Android app (Kotlin + Jetpack Compose)
│   ├── app/src/main/java/com/ktmb/crowdtrend/
│   │   ├── core/          # Models, UI components, utilities
│   │   ├── data/          # Repositories (Live, TransitAlarm)
│   │   └── feature/       # Screen composables (Home, Forecast, Live, Stations, Alarms, Info)

├── docs/             # HTML prototype + generated data
│   ├── data/              # Generated JSON (stations, forecasts, routes)
│   │   ├── by_origin/     # Per-station hourly crowd forecasts (Komuter)
│   │   ├── komuter_utara/ # Per-station forecasts (Komuter Utara)
│   │   ├── ets/           # Per-station forecasts (ETS, not yet used)
│   │   └── intercity/     # Per-station forecasts (Intercity, not yet used)
│   ├── vendor/            # Vendored third-party (gtfs-realtime.proto)
│   └── *.html             # Multi-page HTML prototype
├── scripts/          # Python data pipeline
│   ├── build_data.py      # parquet → JSON
│   ├── build_gtfs_routes.py # GTFS static → route polylines + bbox
│   └── validate_data.py   # Data validation
├── src/              # Cloudflare Worker (GTFS-RT CORS proxy)
├── .github/workflows/    # GitHub Actions (build-data.yml)
└── requirements.txt  # Python deps (pandas, pyarrow)
```

## Development Principles

1. **Android-first** — The primary user-facing app is the Android Compose app in `android/`. HTML prototype is for exploration/rapid prototyping.
2. **Data-driven** — All forecast screens consume processed JSON from `docs/data/`. Data is generated by Python scripts in `scripts/`.
3. **Offline-capable** — Forecast/station data is bundled as assets. Only live vehicle positions require network.
4. **Malaysian context** — State names, station names, service categories, and language (Malay + English) must be correctly handled.

## Code Conventions

### Android (Kotlin/Compose)
- **State**: `StateFlow` in ViewModels, collected as Compose state
- **UI**: Material3 + custom dark theme (`Theme.kt`), glass-morphism cards
- **Navigation**: Bottom nav with `NavHost` (6 tabs: Home, Forecast, Live, Stations, Alarms, Info)
- **Persistence**: DataStore for preferences, bundled JSON for forecast data
- **No DI framework** — manual dependency injection through `AppContainer`
- **File naming**: PascalCase for classes, camelCase for functions
- **Package structure**: `core/` (shared), `data/` (repos), `feature/` (screens)

### Python (data pipeline)
- Pandas + pyarrow for parquet processing
- Output: structured JSON per-origin forecast files
- Scripts are standalone; no package structure

### JavaScript (Cloudflare Worker)
- Single-file worker (`src/index.js`)
- Deployed via Wrangler CLI

### Git
- Branch: `main`
- No merge commits in history (linear)
- Commit messages should follow: `<type>: <description>` (e.g., `fix:`, `feat:`, `chore:`)

## Data Sources

| Source | Location | Purpose |
|--------|----------|---------|
| data.gov.my parquets | `scripts/` input | Ridership data (Komuter, Utara, ETS, Intercity) |
| GTFS static | `scripts/build_gtfs_routes.py` input | Route polylines, station coordinates |
| GTFS-Realtime | API → Cloudflare Worker → Android | Live vehicle positions |

## When Modifying Code

1. **Never touch `docs/data/`** — auto-generated by build pipeline; changes will be overwritten
2. **Android assets** — update bundled JSON alongside any schema changes
3. **Service names** — use consistent casing: Komuter, Komuter Utara, ETS, Intercity
4. **Station names** — match `stations.json` exactly; include state mapping
5. **Crowd levels** — use the enum/scale defined in `CrowdLevel` (1–5)

## Navigation
The app has **6 bottom nav tabs**: Network (Home), Forecast, Live, Stations, Alarms, Info.
- Home is the default start destination
- Stations screen can navigate to Forecast with prefilled origin/destination
- Home screen has quick actions that navigate to Forecast, Live, Alarms

## Transit Alarms Feature
Location-based GPS proximity alarms for approaching stations.
- `TransitAlarmService` — foreground service using `LocationManager` + `LocationListener`
- `TransitAlarmRepository` — persists alarms in app-private JSON file
- Requires `ACCESS_FINE_LOCATION`, `ACCESS_COARSE_LOCATION`, `POST_NOTIFICATIONS` permissions
- Configurable radius: 500m / 1km / 2km
- Station coordinates in `assets/data/station_coords.json` (152 stations)

## Build/Test Commands

# Python data pipeline
cd scripts && python build_data.py
cd scripts && python build_gtfs_routes.py

# Data validation
cd scripts && python validate_data.py
```

## Safety Boundaries

### Do Not Delete
- Gradle files (`*.gradle.kts`, `gradle.properties`, `settings.gradle.kts`)
- AndroidManifest.xml
- Kotlin/Java source files
- App resources (drawables, strings, themes, assets/data/*.json)
- `.github/workflows/` files
- Project documentation (`*.md` files that explain architecture or decisions)
- `docs/vendor/` files
- `scripts/` Python files
- `src/index.js` (Cloudflare Worker)
- `requirements.txt`
- `wrangler.jsonc`

### Do Not Commit
- API keys, tokens, passwords, or secrets of any kind
- Generated build outputs (*.apk, *.aab, build/)
- OS junk files (`.DS_Store`, `Thumbs.db`)
- IDE config (`.idea/` is already gitignored)
- Screenshots, mockup images, ZIP archives
- Node modules, Python venvs, Gradle caches

## Public Repo Hygiene
- This is a **public** repository on GitHub (`abinmo3/ktmb-dashboards`)
- Anyone can see the code, commit history, and documentation
- Bug reports and feature requests may come from external contributors
- Documentation in `*.md` files should be professional and up-to-date
- Commit messages should be descriptive and follow conventions

## Android-Specific Rules
- Minimum SDK, target SDK, and Compose BOM version — check `app/build.gradle.kts`
- Always run `assembleDebug` after Kotlin changes to catch compile errors
- The app must remain buildable at all times
- Do not remove permissions without understanding their usage (especially location + notification perms for TransitAlarms)
- Test on emulator or device before declaring changes complete

## Review Checklist Before Committing
- [ ] No secrets, tokens, or keys in the diff
- [ ] No generated/build files tracked
- [ ] No OS junk or IDE files
- [ ] Android app still builds (`assembleDebug` passes)
- [ ] New files are in the right directories (not root if they belong in a subdir)
- [ ] Documentation updated if architecture/screens/API changed
- [ ] Commit message follows `<type>: <description>` format

[optional body with details]

Types:

  chore:    Maintenance, cleanup, deps

```
