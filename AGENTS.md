# KTMB Pulse — Agent Guidelines

## Project Identity
This is **KTMB Pulse** (Crowd & Trend), a crowd-forecast and live-tracking app for Malaysia's KTM railway network. Two platforms: Android (Kotlin/Compose, primary) and HTML prototype (docs/, secondary).

## Development Principles

1. **Android-first** — The primary user-facing app is the Android Compose app in `android/`. HTML prototype is for exploration/rapid prototyping.
2. **Data-driven** — All forecast screens consume processed JSON from `docs/data/`. Data is generated by Python scripts in `scripts/`.
3. **Offline-capable** — Forecast/station data is bundled as assets. Only live vehicle positions require network.
4. **Malaysian context** — State names, station names, service categories, and language (Malay + English) must be correctly handled.

## Code Conventions

### Android
- **State**: `StateFlow` in ViewModels, collected as Compose state
- **UI**: Material3 + custom dark theme (`Theme.kt`), glass-morphism cards
- **Navigation**: Bottom nav with `NavHost`
- **Persistence**: DataStore for preferences, bundled JSON for forecast data
- **No DI framework** — manual dependency injection through `AppContainer` or similar

### Python (data pipeline)
- Pandas + pyarrow for parquet processing
- Output: structured JSON per-origin forecast files

### GitHub Actions
- Workflow: `.github/workflows/build-data.yml` — runs scripts and commits data updates
- No CI for Android yet (no automated tests)

## When Modifying Code

1. **Never touch `docs/data/`** — auto-generated by build pipeline
2. **Android assets** — update bundled JSON alongside any schema changes
3. **Service names** — use consistent casing: Komuter, Komuter Utara, ETS, Intercity
4. **Station names** — match `stations.json` exactly; include state mapping
5. **Crowd levels** — use the enum/scale defined in `CrowdLevel` (typically 1–5)

## Navigation
The app now has **6 bottom nav tabs**: Network (Home), Forecast, Live, Stations, Alarms, Info.
- Home is the default start destination
- Stations screen can navigate to Forecast with prefilled origin/destination
- Home screen has quick actions that navigate to Forecast, Live, Alarms

## Transit Alarms Feature
Location-based GPS proximity alarms for approaching stations.
- `TransitAlarmService` — foreground service using `LocationManager` + `LocationListener`
- `TransitAlarmRepository` — persists alarms in app-private JSON file
- Requires `ACCESS_FINE_LOCATION`, `ACCESS_COARSE_LOCATION`, `POST_NOTIFICATIONS` permissions
- Configurable radius: 500m / 1km / 2km
- Station coordinates in `assets/data/station_coords.json` (152 stations)

## Data Sources to Know
- `api.data.gov.my` — Malaysian open data portal (primary ridership parquets)
- GTFS-Realtime feed — Live vehicle positions via Cloudflare Worker proxy
- Station, route, and meta JSON — Generated and stored in `docs/data/`

## When Adding Features
- Check `POST_MVP_ROADMAP.md` first — features are assessed for value, effort, dependencies
- Check `DATA_OPPORTUNITIES.md` for extraordinary feature concepts
- Check `current_ui_state.md` for existing UI patterns to match
- Always update `current_ui_state.md` when UI changes

## Deployment
- Android: Build APK via Gradle (`./gradlew assembleDebug` in `android/`)
- HTML prototype: Static site in `docs/` (GitHub Pages compatible)
- Cloudflare Worker: Deploy from `src/` with `wrangler deploy`
